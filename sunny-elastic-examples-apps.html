Generate Seed Indices for Evaluation Data

A prompt that helps you build a curated seed set of Elasticsearch indices for evaluating retrieval quality. Feeds into the Expand Evaluation Dataset workflow to grow coverage without losing intent.

Prompt

This prompt generates a curated set of "seed" Elasticsearch index queries you can use to bootstrap evaluation datasets — the kind of query-document pairs that get fed into retrieval-quality benchmarks, A/B test harnesses, or ELSER fine-tuning loops.

Give it your domain context (a description of what your users typically search for) and a target volume, and you get back structured JSON: realistic queries paired with the index they should hit, plus the relevance grade you'd expect from a perfect retriever. That seed set then becomes the starting point for the Expand Evaluation Dataset From Seed Indices prompt.

What it does

Most evaluation work stalls on the same problem: you have an index, you have users, but you don't have a ground-truth query set that reflects what real users actually ask. This prompt closes that gap by generating queries grounded in the data itself, not made up in a vacuum.

The model reads sample documents from the indices you name, infers the topical surface area, then writes queries that would plausibly hit each cluster of documents — with controlled diversity so you don't end up with twelve paraphrases of the same question.

Inputs

Three fields. Domain description (1–3 sentences on what the index covers). Index names (a comma-separated list). Target volume (how many seed query-document pairs you want, typically 50–500).

Outputs

A JSON array of objects, each containing query, expected_index, expected_doc_id, relevance_grade (0–3 on the standard nDCG scale), and a free-text rationale field explaining why the model picked that pair. The rationale field is the most useful part for QA — you can spot a confused model from a glance through the rationales.

When to use it

At the start of any new search project, before you've shipped to real users and have query logs to mine. Also useful when you've materially changed your index schema and want to refresh your eval set without throwing away the entire harness.

Pair it with Generate Test Cases for Evaluation for adversarial inputs (typos, partial matches, multi-intent queries) once you have the happy-path covered.
