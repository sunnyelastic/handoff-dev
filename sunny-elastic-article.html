You can't whisper at an AI agent

There's a comforting mental model that many of us carried into the age of AI agents, that they'd behave like diligent junior engineers. All you have to do is hand them a codebase, point them at the docs, and they'd read everything, follow best practices, and ask clarifying questions when stuck. However, after running a series of agent steering experiments at Elastic over the past several months, the reality is both messier and more interesting. Agents were integrating with older APIs and were actively making decisions that were contrary to Elastic's recommended best practices without the knowledge of the users.

We tried a lot of things. We modified SDKs. We added warnings to API responses. We restructured skill files. We placed install prompts on CLI login screens and docs pages. Some of these worked remarkably well. Others failed in ways that forced us to rethink our assumptions about how agents process context and, more fundamentally, about what "steering" even means when your user is a language model.

This post walks through what we found, what surprised us, and the two conclusions we think matter most for anyone building developer tools in an agent-first world.

The experiments

A quick sketch of the landscape. Our team ran roughly a dozen experiments aimed at a single question: how do you get an AI agent to use Elasticsearch correctly? "Correctly" here means using current API versions, following integration best practices, and leveraging the Elastic skill. This is a structured set of instructions and references that agents can consume to understand our platform.

The experiments fell into three broad categories:

Passive hints. We tried embedding guidance in places agents might encounter it organically, adding warning hashes in API responses, steering cues in SDK source files, and AGENTS.md files in package directories. The idea was that agents, like good developers, would notice these signals and adjust.

Active prompts. We restructured how skill files were organized (progressive disclosure versus monolithic blobs), added agent-specific sections to CLI help output, and surfaced skill install commands during authentication flows and onboarding.

Distribution plays. We put skills on docs.elastic.co, added install buttons to documentation pages, and ran an onboarding experiment offering users an "AI/agent" path alongside the traditional dashboard flow.

The results split very cleanly. Passive hints failed while active prompts and distribution plays worked, some of them far better than expected. Before we get into the why, it's worth pausing on how unintuitive that split is. Most of our priors said the opposite. We assumed the agents we'd be working with were broadly competent generalists. We assumed they'd read what we put in front of them. We assumed that a well-placed comment in source code would do real work.

None of that was wrong, exactly, but it was the wrong frame. The agents we tested with, even the strongest ones, behaved less like attentive readers and more like rushed contractors who were trying to ship something today. They wanted clear instructions, not breadcrumbs. They wanted a path, not a sign post.

To make the stakes concrete: here is the kind of query an agent needs to construct correctly on a vector index. Get one parameter wrong (wrong field name, missing method_parameters, no _source exclusion) and the response is degraded, slow, or wrong.

GET <INDEX_NAME>/_search
{
  "query": {
    "knn": {
      "<DENSE_VECTOR_FIELD_NAME>": {
        "vector": [...],
        "k": <NUMBER_OF_CANDIDATES>,
        "method_parameters": {
          "ef_search": <NUMBER_OF_CANDIDATES>
        },
        "rescore": {
          "oversample_factor": <OVERSAMPLE>
        },
        "filter": {
          <SOME_FILTER>
        }
      }
    }
  },
  "size": <RESULT_SIZE>,
  "_source": {
    "excludes": [
      "<DENSE_VECTOR_FIELD_NAME>"
    ]
  }
}

Embedding this pattern inline in skill files seemed like the obvious move. It is the right pattern, after all. But as we will see, getting the pattern into the file is not the same as getting the agent to use it.

What failed, and why it matters

The clearest signal from our experiments was this: agents do not really "discover" guidance the way human developers do. When we left structured cues in places we expected agents to look — README files, source comments, response headers — they simply did not look. The cues sat there, perfectly available, and were ignored almost entirely.

This is harder to swallow than it sounds. We've spent two decades building developer ergonomics around the assumption that good engineers will read the room. They'll notice a deprecation warning. They'll click a link in an error message. They'll skim a README before reaching for the API. Agents, in our tests, did none of these things consistently. They had a goal, they had a tool, and they used the tool to make progress on the goal. Anything between those two points was negotiable, and often skipped.

What makes this matter beyond our specific case is that an enormous amount of developer-facing software is structured around passive discovery. SDKs hide their best practices in long-form docs. CLIs assume users will run --help. Error messages link out to runbooks. None of that machinery works the way we expect when the consumer is an agent.

Why passive hints failed

We tried three flavors of passive hint, each with the same disappointing result. We added structured comments to SDK source files marking which methods were preferred over which others. We attached X-Recommended-Endpoint headers to API responses pointing at newer alternatives. And we placed AGENTS.md files in package directories, expecting the agent to find them on first read.

The SDK comments were ignored because agents typically don't scan implementation source when they can read a signature. The response headers were ignored because agents don't read headers unless they have explicit reason to. And the AGENTS.md files were ignored because agents don't grep around looking for documentation. They reach for what they already know.

There's a tidy lesson in here about surface area. If your guidance is not on the path the agent is already walking, the guidance does not exist. It does not matter how clearly it's written, how well-placed it is, or how strongly it's worded. Off-path is invisible.

What worked instead

Active prompts and distribution plays worked, and they worked for the same reason passive hints failed. They put guidance directly into the agent's path of least resistance. A CLI that prints a one-line install hint at login time gets read. A docs page that loads with an install button visible above the fold gets clicked. An onboarding flow that offers an "AI/agent" track at sign-up time gets followed.

None of this is subtle. The win was not from cleverer hints, it was from putting the same hints in places agents could not avoid. The lesson, then, is geometric: position matters more than message.

The two conclusions

We came out of this stretch of work with two takeaways that we think matter beyond our specific platform.

First, treat agents like users with very short attention spans and very long memories. They will skip almost everything you put in front of them on the first pass. But if they do internalize a piece of guidance — usually because they hit it head-on and could not route around it — they will apply that guidance consistently across hundreds of downstream calls. The cost of getting through is high, the payoff is durable.

Second, build for the path, not the periphery. The biggest gains we saw did not come from writing better docs. They came from moving good docs into the few choke points where agents could not skip them. Login flows, CLI help output, onboarding screens, and install buttons on landing pages did most of the work. The long-form material still matters, but it matters most as a place to land after the agent has already been pointed there.

That second one is the harder shift culturally. It asks teams that have spent years polishing their reference docs to also obsess over the half-dozen surfaces that actually route attention. The reference docs are still right. They're just no longer enough.

We're continuing this work and we'll share more as we learn. If you're building developer tools and running similar experiments, we'd love to hear what's working for you and what isn't. The space is moving fast, and the answers are still wide open.
